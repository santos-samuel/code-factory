# RFC Problem Statement Template

*This template is for the first step in the RFC process. It describes the goals of a potential project, provides justification for the work, and demonstrates understanding of the problem.*

*This document:*
- *MUST NOT include: detailed technical architecture, code snippets, project plans*
- *MUST contain ideas and supporting data*
- *MUST be digestible by an experienced engineer without specific domain knowledge in ~30 minutes*
- *SHOULD be 3-4 pages, or 1000-1200 words*

# Overview

*Elevator pitch: what this project will do in 2-3 sentences. Include any historical context not covered in later sections. Assume readers know nothing about the project.*

# Problem Description

*Describe the problem: what it is, why it should be solved, and when. Focus on the pain caused and impact if solved. If upgrading/replacing an existing system, cover that. If new, describe why now.*

# Stakeholders

*Other services, applications, and teams that will:*
- *Use, depend on, or call this application/service*

# Requirements

*What this project must do. Cover:*
- *Use cases with concrete examples*
- *Cost constraints with budget range*
- *Backwards compatibility requirements*
- *Performance requirements with measurable targets*
- *Reliability requirements with SLO targets*
- *Security and compliance requirements*

# Constraints

*Hard limits on the solution space:*
- *Latency, throughput, or storage limits*
- *Staffing and timeline constraints*
- *Compliance requirements (PCI, HIPAA, SOX)*
- *Existing system dependencies that cannot change*

# Out of Scope

*What is explicitly outside the boundaries of this project, and the rationale for each exclusion.*

# Risks and Open Questions

*Known risks and unresolved questions:*
- *For each risk: likelihood, impact, and suggested mitigation*
